# Agent Orchestration Framework

This is a **meta-project**: a multi-agent orchestration framework that uses Claude Code to coordinate PM and Dev agents for automated software development.

## What this repo IS

The deliverable is the orchestration infrastructure:

- `.agents/` ‚Äî Agent identities (`pm.md`, `dev.md`, `architect.md`, `reviewer.md`), orchestrator script, config
- `.claude/commands/` ‚Äî Skills (slash commands) agents use: `gen-spec`, `gen-answer`, `gen-question`, `update-status`, `gen-arch`

## What this repo is NOT

Everything else is **test data** generated by running the agents. These files exist only to validate the framework and must never be committed:

- `docs/` ‚Äî Example domain documentation (input for PM Agent)
- `pm/architecture.md` ‚Äî Example architecture doc generated by Architect Agent
- `pm/specs/` ‚Äî Example specs generated by PM Agent
- `pm/questions/` ‚Äî Example questions/answers from dev‚Üîpm loop
- `pm/tasks/` ‚Äî Example task files created by Dev Agent
- `src/` ‚Äî Example code written by Dev Agent

## Architecture

```
orchestrator.sh
‚îú‚îÄ‚îÄ PM Agent (pm.md)          ‚Äî generates specs, answers questions
‚îÇ   ‚îú‚îÄ‚îÄ /gen-spec             ‚Äî create spec files
‚îÇ   ‚îú‚îÄ‚îÄ /gen-answer           ‚Äî answer dev questions
‚îÇ   ‚îî‚îÄ‚îÄ /gen-question         ‚Äî (used by dev, answered by PM)
‚îú‚îÄ‚îÄ Architect Agent (architect.md) ‚Äî establishes and enforces architecture
‚îÇ   ‚îú‚îÄ‚îÄ /gen-arch             ‚Äî create/update pm/architecture.md
‚îÇ   ‚îî‚îÄ‚îÄ /gen-spec             ‚Äî create corrective specs
‚îú‚îÄ‚îÄ Dev Agent (dev.md)        ‚Äî plans, implements specs, creates PRs
‚îÇ   ‚îú‚îÄ‚îÄ /gen-question         ‚Äî ask PM when blocked
‚îÇ   ‚îî‚îÄ‚îÄ /update-status        ‚Äî transition spec status + sync BACKLOG.md
‚îî‚îÄ‚îÄ Reviewer Agent (reviewer.md) ‚Äî reviews PRs via GitHub
    ‚îî‚îÄ‚îÄ /update-status        ‚Äî transition spec status after review
```

### Spec lifecycle

`backlog ‚Üí in_progress ‚Üí in_review ‚Üí done`

Review modes (`orchestrator.review_mode` in config):
- `agent` ‚Äî Reviewer Agent reviews and merges automatically
- `human` ‚Äî Orchestrator creates PR, human reviews on GitHub
- `hybrid` ‚Äî Reviewer Agent reviews, human merges

## Key commands

```bash
orchestrator.sh pm-seed <prompt>           # Generate initial backlog
orchestrator.sh pm-add <prompt>            # Add spec(s) for a new requirement
orchestrator.sh pm-add-interactive         # Refine a requirement interactively with PM
orchestrator.sh pm-answer-pending          # Answer all pending questions
orchestrator.sh arch-init <prompt>         # Generate initial architecture doc
orchestrator.sh arch-add <prompt>          # Update architecture with new guidelines
orchestrator.sh arch-add-interactive       # Discuss and update architecture interactively
orchestrator.sh arch-review                # Review code compliance with architecture
orchestrator.sh dev-implement SPEC-XXX     # Implement a spec (dev‚Üípm‚Üídev‚Üíreview loop)
orchestrator.sh dev-implement-next         # Auto-pick and implement next eligible spec
orchestrator.sh dev-address SPEC-XXX       # Address review comments on a spec's PR
orchestrator.sh dev-auto                   # Unattended: implement all eligible specs continuously
orchestrator.sh review-pending             # Review all specs in 'in_review' status
```

Stop `dev-auto` with `Ctrl+C` or `touch .agents/.stop` from another terminal.

## Maintenance

After any change to commands, configuration, or architecture, check if `README.md` and `CLAUDE.md` need updating.

## Configuration

`.agents/config.yaml` ‚Äî project-level settings:
- `project.source_dir` ‚Äî where the dev agent writes code (default: `src/`)
- `orchestrator.max_cycles` ‚Äî max dev‚Üípm‚Üídev cycles (default: 3)
- `orchestrator.base_branch` ‚Äî base branch for spec branches / PR target (default: `main`)
- `orchestrator.review_mode` ‚Äî PR review workflow: `agent`, `human`, or `hybrid` (default: `agent`)
- `agents.pm_model` ‚Äî model for PM Agent (default: `opus`)
- `agents.dev_model` ‚Äî model for Dev Agent (default: `opus`)
- `agents.arch_model` ‚Äî model for Architect Agent (default: `opus`)
- `agents.reviewer_model` ‚Äî model for Reviewer Agent (default: `opus`)

## Conventions

- Agent identity files use YAML frontmatter + markdown body (same pattern as specs)
- Skills are Claude Code slash commands in `.claude/commands/`
- The orchestrator invokes agents via `claude -p` with scoped `--allowedTools`
- Both agents have web access (WebSearch, WebFetch) for research
- Architect Agent has no Bash access (read/write only, same as PM)
- PM Agent has no Bash access (read/write only); Dev and Reviewer Agents have Bash for tests/builds/gh CLI
